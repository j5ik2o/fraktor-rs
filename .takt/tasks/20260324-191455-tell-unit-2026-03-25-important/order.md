# `tell(): Unit` 化リファクタリング依頼

更新日: 2026-03-25

> [!IMPORTANT]
> このドキュメントで最優先の指示は「公開 `tell` を Pekko 互換の `Unit` 契約へ揃える」ことです。
> 迷ったら Pekko 互換仕様に立ち返って判断してください。
> `tell` の同期失敗を公開 API 契約として温存しないでください。
> `ask` は別物です。`tell` と同じ失敗モデルへ寄せないでください。
> `tell` を使う関数の仕様も、旧 `Result` 契約への依存があれば推移的に更新してください。
> このドキュメントを更新する場合は、先頭の `IMPORTANT` ブロックと末尾の「最終リマインド」を同時に確認してください。

モデルの性能を最大限に引き出すために、詳細な実装手順はここでは固定しません。  
達成したい設計意図、守るべき制約、完了条件だけを示します。  
具体的な変更順序、局所的な設計判断、影響範囲のさばき方は、コードベースと既存実装を読んだ上で最適化してください。

## 読み方

- まず `IMPORTANT` を読む
- 次に「目的」「設計意図」「守るべき制約」を読む
- 実装中に迷ったら「完了条件」と「最終リマインド」に戻る
- 局所最適ではなく、この文書全体を `tell(): Unit` 化の単一の真実源として扱う

## 目的

`ActorRef::tell` と、その typed/untyped の薄いラッパー群を、Pekko と同じ at-most-once / fire-and-forget の意味へ揃えたいです。  
そのため、公開 API としての `tell` から `Result<(), SendError>` を除去し、`Unit` を返す形へ変更してください。

期待する方向性:

```rust
pub fn tell(&self, message: AnyMessage) {
```

typed 側も同様:

```rust
pub fn tell(&mut self, message: M) {
```

## 設計意図

- この変更は配送保証の変更ではありません
- `tell` は at-most-once / fire-and-forget として扱ってください
- 公開 API から同期的な送信失敗を返さないことが主眼です
- Pekko 互換の意味論を優先してください
- 呼び出し側が「`tell` の失敗を同期的に処理できる」と誤解しない形へ整理してください
- 迷ったら Pekko 互換仕様に立ち返って判断してください

## スコープ

最低限、以下に関係する `tell` 契約を見直してください。

- untyped actor ref
- typed actor ref
- child ref
- recipient trait / interface
- `tell` を直接ラップしている公開 API

## 守るべき制約

- `ask` は別物として扱うこと  
  返信 future / timeout / failure の契約を壊さないでください
- 失敗観測を無言で消さないこと  
  現在の dead letter / log / 観測経路を壊さないでください
- `tell(): Unit` 化と無関係な API 拡張を持ち込まないこと
- `allow` で lint を回避しないこと
- 既存パターンを読んだ上で、一貫した変更にすること

## 判断優先順位

1. 公開 `tell` を `Unit` 契約へ揃える
2. 迷ったら Pekko 互換仕様に寄せる
3. `ask` の契約を壊さない
4. 失敗観測を消さない
5. `tell(): Unit` 化と無関係な改善を混ぜない

## 期待する変更の性質

- `tell` の戻り値だけを理由に `Result` を返していた関数は、必要なら `Unit` に寄せてください
- ただし、その関数に `tell` 以外の失敗要因があるなら、機械的に `Unit` 化せず意味を保ってください
- `tell` の旧契約に依存していた箇所は、表面上のコンパイル修正だけで済ませず、意図を確認してください
- `tell` を直接呼ぶ関数だけでなく、その関数仕様に依存する上位の関数・trait・テストも推移的に見直してください
- 特に actor / remote / cluster / persistence の境界では、観測性と契約の両方を確認してください

## 非目標

- 配送保証の強化
- mailbox / dispatcher の意味論変更
- dead letter の全面再設計
- `ask` の全面 redesign
- `tell(): Unit` 化と無関係な周辺 API の新設

## 完了条件

1. 公開 `tell` 系 API が `Result<(), SendError>` を返さない
2. typed / untyped で `tell` 契約が一貫している
3. `tell` の旧戻り値に依存した call site が適切に整理されている
4. `tell` を使う関数仕様の変更が、上位 API / trait / テストまで推移的に反映されている
5. `ask` 系は壊れていない
6. 失敗観測の経路は残っている
7. テストが更新されている
8. 最後に `./scripts/ci-check.sh ai all` を通す

## 補足

この依頼は「詳細手順の遵守」ではなく「設計意図の達成」を求めます。  
必要なら既存コードを広く読み、局所解ではなく整合的なリファクタリングとして完了させてください。
