# ストラングラーフィグ型モジュール差し替え

抜本的な構造改革が必要なとき、**対象モジュール内を改修し続ける** のではなく、**新モジュールを並行に立て、接続点を付け替え、旧モジュールを退役させる** 戦略を採る。

## なぜ必要か

モジュール `mod_a/` を直接改修しようとすると、`mod_a/*.rs` の既存コードが重力として働き、目的地までたどり着けない:

- ファイル間の相互参照・内部 `pub(crate)` が構造改革の試行錯誤を縛る
- 旧責務と新責務が同居する過渡状態に「compile は通るが設計が濁った」中間コミットが溜まる
- ステップごとに整合を取ろうとして Phase が細切れになり、毎ステップの設計判断が近視眼的になる
- 周辺モジュールからの `use mod_a::...` が増えるほど、境界変更のたびに放射状に diff が広がる

これを避けるため、新モジュール `mod_new_a/`（または最終名 `mod_a_v2/` 等）を **別の場所に独立して構築** し、完成してから接続点だけを一斉に切り替える。

## 基本戦略（4 段階）

```
[Phase 1] 新モジュール新設       mod_a/ はそのまま稼働、mod_new_a/ を別に立てる
[Phase 2] 新モジュール完成       mod_new_a/ 単体でテスト可能な状態まで持っていく
[Phase 3] 接続点の付け替え       mod_a の利用者を mod_new_a に切り替える
[Phase 4] 旧モジュール退役       mod_a/ を削除する
```

各 Phase は独立コミット / 独立 PR が望ましい。Phase 3 と Phase 4 を同一 PR にするのは参照 0 件確認が容易な場合のみ。

## 適用判断

| 判定 | 条件 |
|------|------|
| ✅ 採用 | 既存モジュールの責務境界そのものが誤っており、in-place 改修では責務が分離できない |
| ✅ 採用 | 1 ファイル数百行級の神モジュールで、抽出を試みると毎回相互依存でロールバックする |
| ✅ 採用 | 公開 API の契約を変えたい（型・trait 名・引数形状）が、旧 API 利用者が多く一括移行は困難 |
| ⚠️ 要検討 | 命名変更と軽いファイル分割だけで済む（`plan-package-migration` の範囲で足りる可能性） |
| ❌ 非採用 | バグ修正・小規模改善（in-place が正解） |
| ❌ 非採用 | 新モジュールと旧モジュールを恒久併存させる計画（それは差し替えではなく分岐） |

## 命名と配置

### 新モジュールの仮名

- 作業期間中のみ使う仮名として `new_` プレフィックスは許容（`mod_new_a/`、`actor_cell_v2/` など）
- **リリース時点で仮名を残さない**。Phase 4 で旧モジュールを削除したあと、**同一 PR または直後の PR** で仮名を最終名にリネームする
- 仮名のまま merge してリリース境界をまたぐと、利用者から見て「どちらが本物か」が永続的に不明瞭になる

### 最終名を先に使う場合

新モジュールに最終名を先取りできるケース（旧が即座に退役予定で、共存期間が短い）:

- 旧を `mod_a_legacy/` にリネームして新を `mod_a/` にする
- この場合 Phase 1 と Phase 3 が近接し、Phase 2 の実装中に `mod_a_legacy` から `mod_a` への参照切替も同時進行する
- リスクはやや高いが、仮名リネームの手間は省ける

どちらを採るかは、新モジュール完成までの期間と旧利用者の多さで判断する。**迷ったら仮名先行 (`new_` プレフィックス) 方式**。

## 接続点の付け替え手順

Phase 3 は差し替えの中核。**接続点を漏れなく洗い出してから一括で切り替える**。

### 接続点の種類

| 種類 | 検索方法 |
|------|----------|
| `use mod_a::...` / `crate::mod_a::...` | `rg -n 'use .*(^|::)mod_a(::|;)'` |
| 親 `mod_a;` 宣言 | `rg -n '^pub\s+mod\s+mod_a|^mod\s+mod_a'` |
| `pub use mod_a::...` (再エクスポート) | `rg -n 'pub use .*mod_a::'` |
| trait 実装の参照 | `rg -n 'impl .*for mod_a::'` |
| ドキュメント・コメント内の型名参照 | `rg -n 'mod_a::\w+' --type md` |
| Cargo.toml feature / path | `rg -n 'mod_a' Cargo.toml modules/*/Cargo.toml` |

### 付け替えの順序

1. **利用者ゼロのリーフから切り替え**: テストファイル、`bin/`、`examples/` → 次に単独利用モジュール → 最後に多用されるモジュール
2. **各切替コミットで compile + test 通過を維持**: 一括置換で一時的に双方参照する状態は許容するが、PR 内で必ず収束させる
3. **`pub use` 互換シムは置かない**: `mod_a` 側から `pub use mod_new_a::*` で橋渡しするのは一見便利だが、Phase 4 で剥がすコストが復活するだけ。直接 import を書き換える

## 退役判定

Phase 4 に進める条件:

- `rg -n '(^|::)mod_a(::|;|\s|$)'` が **テストと CHANGELOG を含めて 0 件**
- `cargo check` / `cargo test` が **mod_a/ 物理削除後に通る**
- 削除した公開 API について、`docs/` 配下に「旧名」への参照が残っていない

上記を満たさない限り Phase 4 は ABORT する。**「ほぼ使われていないから消す」は禁止**（あとで `git log` を読む人に旧名検索のヒットを残すのも避ける）。

## in-place 改修との使い分け（必須判断）

plan ステップで「リファクタリング」と呼ぶ作業は、次のどちらかに分類する:

| 種別 | 使う指示書 | 戦略 |
|------|-----------|------|
| in-place 改修 | `plan-refactoring` / `plan-package-migration` | 既存 `mod_a/` 内で責務分離・命名是正 |
| 差し替え移行 | 本ナレッジ + `plan-refactoring` 併用 | 新モジュールを立てて接続点を付け替える |

**判定フロー**:

```
1. 対象モジュールの責務境界そのものを変えるか？
   ├─ No → in-place 改修（本ナレッジは不要）
   └─ Yes → 次へ

2. in-place で責務分離を試みて compile/test を維持できる見通しがあるか？
   ├─ Yes → in-place 改修で挑戦。詰まったら次へ
   └─ No → 次へ

3. 新モジュールを恒久的に並走させる計画か？
   ├─ Yes → 差し替えではなく分岐。設計をやり直す
   └─ No → 差し替え移行を採用（本ナレッジ適用）
```

## 既存ルールとの整合

- **`legacy-code-temporary-usage.md`**: 旧モジュール `mod_a/` は「短期の過渡状態」を表す。Phase 4 で必ず削除する。共存を恒久化しない
- **`package-design-principles.md` 段階移行のガードレール**: `pub use` 互換シム・`#[path = "..."]`・二重維持は禁止。本ナレッジも同じ制約下で動く
- **`pekko-porting`**: Pekko parity 実装でも、`core/kernel/{domain}` 全体を差し替えるほどの構造改革が必要な場合は本戦略を適用する

## チェックリスト

差し替え移行を採用する plan レポートには以下を必ず含める:

- [ ] 採用理由（なぜ in-place では無理か）
- [ ] 新モジュール名と仮名ポリシー（`new_` 先行か、最終名先行か）
- [ ] 各 Phase の境界（単一 PR 境界と一致させる）
- [ ] 接続点一覧（上記「接続点の種類」テーブルの各行を実行した結果）
- [ ] 退役判定条件（`rg` で 0 件を確認するコマンド）
- [ ] 仮名リネームの最終 PR 位置（`new_` 先行の場合）
- [ ] 並走期間中のコミット境界（一時的な双方参照状態を含む commit は許容範囲か）

## アンチパターン

- **仮名を残したままリリース**: `new_` プレフィックスが公開 API に露出する
- **旧モジュールを `pub use` でラップ**: 差し替えではなく互換層の積み増し
- **Phase を跨いで参照を残す**: Phase 4 まで進んだのに `use mod_a` が残っている
- **新旧両方にバグ修正を適用**: 並走期間が長引くと、どちらが正準か不明になる。並走期間は**短く**保つ
- **新モジュールに旧モジュールのテストをそのままコピー**: 責務境界を変えるのが目的なので、テストも新しい契約で書き直す
