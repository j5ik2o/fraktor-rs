# パッケージ設計の原則（fraktor-rs 向け要約）

パッケージ／モジュール構成の判断に使う。詳細は `.agents/skills/package-design` / `refactoring-packages` を参照。

## 核心

配置は「ソースの整理」ではなく、**変更の波をどこで止めるか** と **依存の向きをどう制御するか** の設計である。

## 分割の出発点

- 分割の軸は「処理手順」ではなく **変化しそうな設計決定**（Parnas の情報隠蔽）
- まず「何が変わるか」「変わったときどこまで巻き込んでよいか」を列挙する

## 分割軸の例

| 軸 | 向くケース |
|----|------------|
| 機能（vertical） | チーム境界・変更が縦に閉じる |
| レイヤ（技術層） | 小規模・導入初期（1 変更が全層に散りやすい点に注意） |
| 責務（変更理由） | CCP（共通閉包）に沿う |
| API 境界 | ライブラリ公開面の最小化 |

## パッケージング原則（チェック用）

| 原則 | 確認 |
|------|------|
| 高凝集 | 同一パッケージ内が単一目的に収束しているか |
| 低結合 | パッケージ間依存が最小か |
| ADP | 循環依存がないか |
| SDP | 不安定なものが安定側に依存していないか |
| SAP | 安定側は十分に抽象的か |
| CCP | 同じ変更理由のものが同じパッケージに閉じているか |

## 構造品質ゲート（approve の十分条件として使う）

以下は「問題がない」だけではなく、「改善した」と言うための最低条件。

- **高凝集**: 代表的なグループについて、変更理由を1文で説明できる
- **低結合**: 不自然な双方向依存、三角依存、逆流依存がない
- **フラット配置解消**: root 直下または親階層の dumping ground が減っている、または明確な設計理由がある
- **配置予測可能性**: 同種の責務を1つ追加するとき、置き場が一意に決まる

次の状態は approve してはならない:

- Phase の移設作業だけ完了し、上位の dumping ground がそのまま
- private 化や shim 削除だけで、責務境界の改善を説明できない
- before/after の構造メトリクスがなく、改善量を示せない

## 循環依存の解消の考え方

1. **依存性逆転**: 抽象（trait）を安定側へ、実装を不安定側へ
2. **共有抽出**: 共通型・共通ロジックを第三のモジュールへ
3. **統合**: 実質同一責務ならパッケージをまとめる

## アンチパターン（検出したら指摘候補）

- **神モジュール**: 単一ファイルに複数責務が集中
- **過度なフラット配置**: 小型ディレクトリが同階層に乱立し、発見可能性が下がる
- **類似命名の重複**: `foo` / `foos` のように意味が重なる隣接モジュール
- **型の機械的複製**: アリティ違いだけの型がディレクトリを量産（マクロ・ジェネリクス・生成の検討）
- **雑多な util / common**: 関連性の低い要素の寄せ集め
- **抽象の漏れ**: 内部型が公開 API に漏れる

## 段階移行のガードレール

パッケージリファクタリングの完了は、実際のファイル配置・`mod` 配線・import が新構造に収束していることを意味する。旧構造を温存したまま、見かけ上だけ新構造にすることは完了ではない。

### 禁止

- `#[path = "..."]` で旧ファイルを別モジュールにぶら下げる
- `include!` 等で実ファイル配置を隠す
- 旧モジュールから新モジュールを `pub use` するだけの互換ラッパーを残す
- 同一責務の旧パスと新パスを、同一 Phase 完了時点で併存させる
- import を更新せず、親モジュールや crate root の再エクスポートだけで「移動済み」に見せる
- コンパイル維持のための暫定互換層を残したまま Phase 完了扱いにする

### 許容

- crate root または設計で明示した公開境界での `pub use`。ただし、公開 API の形を定義する目的に限る。旧内部パス互換の維持目的では使わない。

### 行き詰まったとき

1. no-shim で進めるとコンパイルが壊れるなら、Phase をさらに小さく分割する
2. それでも無理なら `plan` または `design` に戻す
3. 安全策は互換レイヤの追加ではなく、変更単位の縮小である

## Rust / fraktor-rs 固有

- **2018 パス方式**: `mod.rs` は使わない。`foo.rs` + `foo/` が基本
- **1 公開型 1 ファイル**: Dylint `type-per-file` 準拠。ディレクトリ移動時は `mod` 宣言と再エクスポートを一貫させる
- **クレート分離**: `modules/{domain}-core/` は原則 `no_std` + Sans I/O（純粋ロジック層、ポート定義）。`modules/{domain}-adaptor-std/` は std/tokio 依存のアダプタ。依存は `{domain}-adaptor-std → {domain}-core` の片方向。構造変更でこの向きを壊さない
  - **AI は例外を実装してはならない**: `{domain}-core` のプロダクトコードに std 依存を新規導入することは AI の判断では禁止。既存コードに std 依存が残っている箇所は過去に **人間の指示** で入った例外であり、AI が同じ理由で拡大してはならない
  - **例外追加には人間の明示指示が必須**: 新たに std 依存を入れたい場合は、必ず人間に相談してから実装する
  - **テストコードは std 依存 OK**: `{domain}-core/**/tests.rs` や `tests/` では std を使ってよい（Sans I/O 制約はプロダクトコードのみに適用）
- **参照実装**: `references/pekko` の `stream` は `scaladsl` / `impl` / `stage` 等で層が分かれる。機械生成（boilerplate）はテンプレートで一括生成する例がある（例: `FanInShapeN`）

## リファクタリング時の優先度（目安）

1. 循環依存（ビルド・保守の致命傷）
2. 神モジュール・変更影響の広がり
3. 散在・ネスト過多による認知負荷
4. 命名・重複による誤読
