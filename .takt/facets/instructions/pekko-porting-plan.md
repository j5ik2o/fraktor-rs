Pekko互換仕様のRust実装をバッチ単位で計画してください。

## やらないこと (Do Not)
- `cargo check` / `cargo build` / `cargo test` など、ビルドを伴うコマンドを実行しないこと。このムーブメントはビルド権限がなく `Operation not permitted` で失敗する。ビルド検証は `fix` / `implement` ムーブメントの責務。

**注意:** Previous Responseがある場合は前バッチの実装結果であるため、
その内容を踏まえて次バッチの選定・計画を行ってください。

## バッチ選定（最初に実行）

1. **ギャップ分析ドキュメントの確認**
   - ユーザーのタスク指示で指定されたギャップ分析ドキュメント（`docs/gap-analysis/*.md`）を読む
   - 前回のバッチで実装済みの項目を Report Directory 内の過去レポート（`coder-scope.md`）から特定する
   - 未実装タスクのうち、関連性の高いものをまとめて今回のバッチとして選定する

2. **バッチサイズの判断基準**
   - trivial/easy: 3〜5項目をまとめてよい
   - medium: 1〜2項目
   - hard: 1項目のみ
   - 依存関係がある項目は同一バッチに含める
   - **並行実装を考慮**: バッチ内の項目が並行実装可能かを判定する（後述）

3. **完了判定**
   - ギャップ分析の対象範囲（ユーザーが指定した Phase やカテゴリ）のすべてが実装済みなら「全タスク完了」
   - まだ残りがあれば「未実装タスクあり」としてバッチを選定する

## バッチ内タスクの計画

選定した各タスクについて以下を実施する。

1. **Pekko参照実装の読み込み**
   - 対象機能のPekkoソースを `references/pekko/` から実際に読む
   - 対象の公開API（trait, class, object, メソッド）を列挙する
   - protoactor-go（`references/protoactor-go/`）に対応する実装があれば併せて確認する

2. **fraktor-rs の現状把握**
   - 対応する `modules/{name}/` の既存実装を確認する
   - 既存の型・メソッドとPekkoのAPIを突き合わせて差分を特定する
   - **要件ごとに「変更要/不要」を判定し、「不要」の場合は現行コードの該当箇所（ファイル:行）を根拠として示す。根拠なしの「既に正しい」は禁止**

3. **Rust変換設計**
   - ナレッジのScala→Rust変換ルールに従ってAPI設計を決定する
   - 型パラメータ（メッセージ型 M 等）の配置を決める
   - `&self` vs `&mut self` をCQS原則に基づいて決定する
   - no_std/std分離の方針を明記する

4. **ファイル構成の決定**
   - 1ファイル1公開型ルールに従ってファイル配置を決める
   - 既存のモジュール構造を壊さない配置にする
   - テストファイルの場所も計画に含める

5. **影響範囲の特定**
   - 既存のコードへの変更が必要な箇所を列挙する
   - 後方互換性は不要（破壊的変更OK）だが、変更箇所は明記する

6. **実装アプローチの決定**
   - 関連するスキルを探索する
   - 段階的に実装できる場合はステップを分ける
   - Dylint lintに違反しない設計であることを確認する

**小規模タスクの判断基準:**
- 1-2ファイルの変更のみ
- 新規型の追加が1-2個のみ
- 設計判断が不要

小規模タスクの場合はRust変換設計セクションを簡略化してよい。

## 並行実装可能性の判定（team_leader 向け）

バッチ内の各タスクについて、以下の基準で並行実装可能性を判定する。
implement ムーブメントでは team_leader が最大3つのサブタスクに並行分解するため、
この情報が分解の品質に直結する。

### 判定基準

| 条件 | 判定 |
|------|------|
| 変更対象ファイルが完全に異なる | ✅ 並行可能 |
| 同一ファイルの異なる impl ブロックへの追加 | ⚠️ 条件付き並行可能（末尾追加なら可） |
| 同一ファイルの同一箇所を変更 | ❌ 逐次実行 |
| タスク B がタスク A の出力型に依存 | ❌ 逐次実行（A→B の順） |
| 共通の新規型を導入してから各タスクが使う | ❌ 逐次実行（共通型→個別タスク） |

### レポートへの記載方法

各タスクに以下の属性を付記する:
- **並行グループ**: 同じグループのタスクは並行実行可能
- **変更対象ファイル**: 競合の有無を明示
- **依存関係**: 「なし」または「タスクXの完了後」

例:
```
タスク1: Sink.never [並行グループ A] [変更: sink.rs] [依存: なし]
タスク2: UniformFanOutShape [並行グループ A] [変更: shape/ 新規] [依存: なし]
タスク3: SubstreamCancelStrategy [並行グループ A] [変更: 新規 + flow.rs] [依存: なし]
```

## レポートに含める内容

- 今回のバッチで実装するタスク一覧（ギャップ分析の該当行を引用）
- 全体の進捗状況（実装済み / 今回バッチ / 残り）
- 各タスクの計画詳細
- **並行実装マップ**: どのタスクが並行可能で、どの順序制約があるか
