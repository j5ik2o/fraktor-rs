## Pekko porting 固有の方針

Pekko互換仕様のRust実装をバッチ単位で計画してください。

目標は **Pekko の契約意図を Rust / fraktor-rs の設計原則を壊さずに再表現すること** である。
Pekko に似せること自体を目的化してはならない。

Previous Response がある場合、この workflow では前バッチの実装結果である可能性が高い。
本家 `plan` の差し戻し/replan 前提だけで扱わず、前バッチの完了内容を踏まえて次バッチの選定・計画を行うこと。
ただし Previous Response が明確に差し戻しや再計画要求を示している場合は、その内容を優先する。

各提案項目は必ず次のいずれかで分類すること。

- **採用**: Pekko の契約を Rust でもほぼ同じ意味で表現できる
- **翻訳**: 契約意図は維持するが、Rust では別の形で表現すべき
- **非採用**: JVM / Java 依存が強く、Rust に持ち込むと設計を悪化させる

## やらないこと

- `cargo check` / `cargo build` / `cargo test` など、ビルドを伴うコマンドを実行しないこと。このステップはビルド権限がなく `Operation not permitted` で失敗する。ビルド検証は `fix` / `implement` ステップの責務。
- バッチ選定と実装計画を作成できる場合は、`要件が不明確、情報不足` を選ばないこと。
- `確認事項` セクションは、計画を確定できない真の blocker がある場合のみ書くこと。単なる追加提案、設計メモ、将来の改善案は `スコープ外` または `実装ガイドライン` に書くこと。
- `docs/gap-analysis/*.md` と直前の `coder-scope.md` / `coder-decisions.md` から合理的に推論できる事項は自己解決すること。
- wrapper / alias / fallback で gap を埋める計画を立てないこと。
- no-op / placeholder のまま public API を置く計画を立てないこと。
- Rust で不自然な Java / JVM 流 DSL を、そのまま移植対象にしないこと。

## バッチ選定

1. ユーザーのタスク指示で指定されたギャップ分析ドキュメント（`docs/gap-analysis/*.md`）を読む。
2. 前回のバッチで実装済みの項目を Report Directory 内の過去レポート（`coder-scope.md`）から特定する。
3. 未実装タスクのうち、関連性の高いものを今回のバッチとして選定する。
4. ギャップ分析の対象範囲（ユーザーが指定した Phase やカテゴリ）のすべてが実装済みなら「全タスク完了」とする。

バッチサイズは固定せず、難易度・依存関係・同時に触るファイル数から「ちょうどよい」粒度を選ぶ。
以下は上限や義務ではなく、判断の目安として扱うこと。

- trivial/easy: 3〜5項目をまとめてもよい
- medium: 1〜2項目を目安にする
- hard: 原則1項目のみ
- 依存関係が強い項目は、分けると不自然なら同一バッチに含める

## Pekko / fraktor-rs 照合

- 対象機能のPekkoソースを `references/pekko/` から実際に読む。
- 対象の公開API（trait, class, object, メソッド）を列挙する。
- protoactor-go（`references/protoactor-go/`）に対応する実装があれば併せて確認する。
- 対応する `modules/*-core*` / `modules/*-adaptor-std` の既存実装を確認する。
- 要件ごとに「変更要/不要」を判定し、「不要」の場合は現行コードの該当箇所（ファイル:行）を根拠として示す。

## Rust変換設計

- Scala→Rust変換ルールに従ってAPI設計を決定する。
- 型パラメータ（メッセージ型 M 等）の配置を決める。
- `&self` vs `&mut self` をCQS原則に基づいて決定する。
- no_std/std分離の方針を明記する。
- `public API` と `internal implementation` の境界を壊さないこと。
- 1ファイル1公開型ルールに従い、既存のモジュール構造を壊さない配置にする。

## 並行実装可能性

implement ステップでは team_leader が最大3つのサブタスクに並行分解するため、各タスクに次の属性を付記する。

- **並行グループ**: 同じグループのタスクは並行実行可能
- **変更対象ファイル**: 競合の有無を明示
- **依存関係**: 「なし」または「タスクXの完了後」
