Pekko互換仕様のRust実装をバッチ単位で計画してください。

## 最重要方針

目標は **Pekko の契約意図を Rust / fraktor-rs の設計原則を壊さずに再表現すること** である。
Pekko に似せること自体を目的化してはならない。

各提案項目は必ず次のいずれかで分類すること:
- **採用**: Pekko の契約を Rust でもほぼ同じ意味で表現できる
- **翻訳**: 契約意図は維持するが、Rust では別の形で表現すべき
- **非採用**: JVM / Java 依存が強く、Rust に持ち込むと設計を悪化させる

## やらないこと (Do Not)
- `cargo check` / `cargo build` / `cargo test` など、ビルドを伴うコマンドを実行しないこと。このステップはビルド権限がなく `Operation not permitted` で失敗する。ビルド検証は `fix` / `implement` ステップの責務。
- バッチ選定と実装計画を作成できる場合は、`要件が不明確、情報不足` を選ばないこと。
- `確認事項` セクションは、計画を確定できない真の blocker がある場合のみ書くこと。単なる追加提案、設計メモ、将来の改善案は `確認事項` に入れず、`スコープ外` または `実装ガイドライン` に書くこと。
- `docs/gap-analysis/*.md` と直前の `coder-scope.md` / `coder-decisions.md` から合理的に推論できる事項は自己解決すること。
- wrapper / alias / fallback で gap を埋める計画を立てないこと
- no-op / placeholder のまま public API を置く計画を立てないこと
- Rust で不自然な Java / JVM 流 DSL を、そのまま移植対象にしないこと

**注意:** Previous Responseがある場合は前バッチの実装結果であるため、
その内容を踏まえて次バッチの選定・計画を行ってください。

## バッチ選定（最初に実行）

1. **ギャップ分析ドキュメントの確認**
   - ユーザーのタスク指示で指定されたギャップ分析ドキュメント（`docs/gap-analysis/*.md`）を読む
   - 前回のバッチで実装済みの項目を Report Directory 内の過去レポート（`coder-scope.md`）から特定する
   - 未実装タスクのうち、関連性の高いものをまとめて今回のバッチとして選定する

2. **バッチサイズの判断基準**
   - trivial/easy: 3〜5項目をまとめてよい
   - medium: 1〜2項目
   - hard: 1項目のみ
   - 依存関係がある項目は同一バッチに含める
   - **並行実装を考慮**: バッチ内の項目が並行実装可能かを判定する（後述）

3. **完了判定**
   - ギャップ分析の対象範囲（ユーザーが指定した Phase やカテゴリ）のすべてが実装済みなら「全タスク完了」
   - まだ残りがあれば「未実装タスクあり」としてバッチを選定する

## 既知スタブとの突合（必須）

バッチ選定前に `.takt/facets/knowledge/stub-elimination.md` を必ず読み、対象モジュール (actor / stream / remote / cluster / persistence) に該当する既知スタブをバッチ候補に優先採用する。

1. **stub-elimination.md を読む**
   - 対象モジュールに該当するスタブ一覧を抽出する
   - 各スタブについて「現状実装」「Pekko セマンティクス」「必要な変更」を把握する

2. **バッチ候補への反映**
   - 既知スタブが当該 Phase / カテゴリの対象範囲に含まれる場合、新規実装ではなく **既存 fake-impl の修正** としてバッチに組み込む
   - 「未実装」ではなく「スタブ是正」として扱う。gap-analysis 側で実装済み扱いになっていても、`stub-elimination.md` に残っていれば本ステップで捕捉する
   - 対象外 Phase のスタブはバッチに入れなくてよいが、レポートには「既知スタブ（対象外 Phase）」として列挙する

3. **レポートに必須項目として記載**
   - セクション名: `既知スタブ対応`
   - 各対応タスクについて以下を明記する:
     - タスク名
     - `stub-elimination.md` の該当グループ（例: Timer系 / Substream系）
     - 現状実装（ファイル:行と当該コード片の引用）
     - Pekko 参照パス（ファイル:行）
     - 目標セマンティクス（Pekko 契約を Rust でどう再表現するか）

4. **ファクトチェック（必須）**
   - `stub-elimination.md` の記述は過去時点の手書きリストであり、既に解消済みの項目を含む可能性がある
   - 該当 API の **実際の実装を必ず読み直し**、以下のいずれかを実施する:
     - 記述が古い場合: 本ファイル更新を同一バッチの成果物に含める
     - 記述が現状と一致する場合: バッチ候補に含める
     - 既に是正済みの場合: レポートに「既に正しく実装済み。stub-elimination.md の該当行は次回更新で除去」と明記する

## バッチ内タスクの計画

選定した各タスクについて以下を実施する。

1. **Pekko参照実装の読み込み**
   - 対象機能のPekkoソースを `references/pekko/` から実際に読む
   - 対象の公開API（trait, class, object, メソッド）を列挙する
   - protoactor-go（`references/protoactor-go/`）に対応する実装があれば併せて確認する

2. **fraktor-rs の現状把握**
   - 対応する `modules/{name}/` の既存実装を確認する
   - 既存の型・メソッドとPekkoのAPIを突き合わせて差分を特定する
   - **要件ごとに「変更要/不要」を判定し、「不要」の場合は現行コードの該当箇所（ファイル:行）を根拠として示す。根拠なしの「既に正しい」は禁止**

3. **Rust変換設計**
   - ナレッジのScala→Rust変換ルールに従ってAPI設計を決定する
   - 各項目について `採用 / 翻訳 / 非採用` のいずれかを明記する
   - 型パラメータ（メッセージ型 M 等）の配置を決める
   - `&self` vs `&mut self` をCQS原則に基づいて決定する
   - no_std/std分離の方針を明記する
   - `public API` と `internal implementation` の境界を壊さないこと

4. **ファイル構成の決定**
   - 1ファイル1公開型ルールに従ってファイル配置を決める
   - 既存のモジュール構造を壊さない配置にする
   - テストファイルの場所も計画に含める

5. **影響範囲の特定**
   - 既存のコードへの変更が必要な箇所を列挙する
   - 後方互換性は不要（破壊的変更OK）だが、変更箇所は明記する

6. **実装アプローチの決定**
   - 関連するスキルを探索する
   - 段階的に実装できる場合はステップを分ける
   - Dylint lintに違反しない設計であることを確認する

**小規模タスクの判断基準:**
- 1-2ファイルの変更のみ
- 新規型の追加が1-2個のみ
- 設計判断が不要

小規模タスクの場合はRust変換設計セクションを簡略化してよい。

## 並行実装可能性の判定（team_leader 向け）

バッチ内の各タスクについて、以下の基準で並行実装可能性を判定する。
implement ステップでは team_leader が最大3つのサブタスクに並行分解するため、
この情報が分解の品質に直結する。

### 判定基準

| 条件 | 判定 |
|------|------|
| 変更対象ファイルが完全に異なる | ✅ 並行可能 |
| 同一ファイルの異なる impl ブロックへの追加 | ⚠️ 条件付き並行可能（末尾追加なら可） |
| 同一ファイルの同一箇所を変更 | ❌ 逐次実行 |
| タスク B がタスク A の出力型に依存 | ❌ 逐次実行（A→B の順） |
| 共通の新規型を導入してから各タスクが使う | ❌ 逐次実行（共通型→個別タスク） |

### レポートへの記載方法

各タスクに以下の属性を付記する:
- **並行グループ**: 同じグループのタスクは並行実行可能
- **変更対象ファイル**: 競合の有無を明示
- **依存関係**: 「なし」または「タスクXの完了後」

例:
```
タスク1: Sink.never [並行グループ A] [変更: sink.rs] [依存: なし]
タスク2: UniformFanOutShape [並行グループ A] [変更: shape/ 新規] [依存: なし]
タスク3: SubstreamCancelStrategy [並行グループ A] [変更: 新規 + flow.rs] [依存: なし]
```

## レポートに含める内容

- 今回のバッチで実装するタスク一覧（ギャップ分析の該当行を引用）
- 全体の進捗状況（実装済み / 今回バッチ / 残り）
- 各タスクの計画詳細
- **並行実装マップ**: どのタスクが並行可能で、どの順序制約があるか
- **採用/翻訳/非採用の判定**: 各タスクの設計判断根拠

## ステータス判定の明示

レポート末尾に必ず以下のいずれか1行を、そのまま書くこと。

- `判定: 未実装タスクあり（バッチ選定完了）`
- `判定: 全タスク完了（ギャップ分析の対象範囲をすべて実装済み）`
- `判定: ユーザーが質問をしている（実装タスクではない）`
- `判定: 要件が不明確、情報不足`

通常の実装継続ケースでは、原則として 1 行目を選ぶこと。
