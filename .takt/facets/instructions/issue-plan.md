# GitHub Issue 実装計画作成

複数の GitHub Issue を実装可能な作業計画に分解してください。

## 手順

1. User Request とともに github issue 番号一覧（`#123` 形式）を抽出する
   - 取得対象 issue は `coderabbit` または `bug` ラベルを持つものに限定する
   - issue 情報が不足する場合は `gh issue view <issue-number>` で補完する
2. issue ごとに「すでに解決済みか」をコード状況とテスト状況で検証する
   - 既存コードに該当修正があるか、関連テストの期待値が満たされるかを確認する
   - 解決済みと判断した場合は、以下を実施する
     - `gh issue comment <issue-number> -b "<日本語コメント>"`
     - `gh issue close <issue-number>`
     - コメント例（日本語・簡潔）: `現状のコードで受け入れ条件を満たしていることを確認しました。対応不要のためCloseします。`
3. issue ごとに情報不足かどうかを判定する
   - 受け入れ条件・再現手順・影響範囲のいずれかが不足している場合は情報不足とする
   - 情報不足と判断した場合は、以下を実施する
     - `gh issue comment <issue-number> -b "<日本語コメント>"`
     - コメント例（日本語・具体）: `受け入れ条件と再現手順が不足しているため判断できません。必要情報（再現手順/期待結果/影響範囲）の追記をお願いします。`
4. issue ごとに目的・制約・受け入れ条件を定義する（解決済み/情報不足は簡潔でよい）
5. issue ごとに実装対象ファイルと非対象（スコープ外）を明確化する
6. issue ごとに完了順序を決める
7. issue ごとに仮のコミットメッセージ案を作る（解決済み/情報不足は不要）
8. issue ごとに `./scripts/ci-check.sh all` 実行タイミングを計画に含める（解決済み/情報不足は不要）
9. 事前にコミット方針を明記する（Issue単位で1コミット以上）

## 重要

- 仕様にない拡張は提案しない（YAGNI）
- 不明点は「前提不足」として明記する
- 受け入れ条件が定義できない issue は「情報不足」として扱い、ABORT せずスキップする
- 解決済み issue は実装対象から除外し、計画に「解決済み」と明記する
- 情報不足 issue は実装対象から除外し、計画に「情報不足」と明記する
- コミットメッセージは Conventional Commits かつ英語であることを計画段階で明記する
- コミットメッセージ案は `(#<issue-number>)` を末尾につける
- 各 issue 完了時点で `./scripts/ci-check.sh all` を実行する前提を明記する
