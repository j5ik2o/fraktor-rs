# パッケージ構造の実装（移行）

Report Directory 内の **`plan.md`**・**`structure-design.md`** を一次情報として、**当該 Phase のみ** ファイル移動とモジュール宣言・import の更新を行う。

Piece Context に示された Report Directory 内のファイルのみ参照すること。他のレポートディレクトリは検索・参照しない。

## 原則

- **Phase をまたがない**（plan に書かれた Phase のみ）
- 各ステップのあと **コンパイル可能な状態** を維持する。no-shim で維持できないなら、Phase が大きすぎるので `plan` / `design` に戻す
- **ロジック変更は最小**。パッケージ移行が主目的のとき、挙動変更は plan に明示されていない限り行わない
- **後方互換・移行用の暫定レイヤを持ち込まない**
- fraktor-rs 規約: `core`/`std` 境界、1 ファイル 1 公開型、Dylint、`mod.rs` 不使用

## 禁止事項

- `#[path = "..."]`
- `include!`
- 旧モジュールから新モジュールへの `pub use` / `pub type` / wrapper mod だけで旧パスを残すこと
- 実ファイルを移動せず、親モジュールや crate root の再エクスポートだけで移動済みに見せること
- 同一責務の旧パス / 新パスを同一 Phase 完了時点で併存させること
- 実装未着手のままレポートだけで完了扱いにすること

## やること

1. `plan.md` の手順に従い、**実ファイル移動**・`mod` 追加／削除・必要最小限の `pub use` を実施する
2. 正準パスが確定した責務は、同一 Phase 内で **旧 `mod` / 旧ファイル / 旧ディレクトリを削除** する
3. ワークスペース全体で **旧パス参照を grep** し、import を更新する
4. `pub use` を残す場合は、`structure-design.md` で許可された公開境界に限定し、`coder-decisions.md` に理由を書く
5. 公開 API を変える場合は、タスク／plan の意図に沿っているか確認する（無断で公開範囲を広げない）
6. no-shim で進められないと判明したら、その場で止めて `plan` / `design` に戻る。互換レイヤを足して通してはならない

## ビルド・テスト（必須）

**やらないこと:**

- `cargo` を直接実行しない

**必ず `./scripts/ci-check.sh` 経由:**

1. 変更したクレートに応じて Dylint を実行する（例: `./scripts/ci-check.sh ai dylint -m stream` またはタスク指定のモジュール）
2. `./scripts/ci-check.sh ai std`（またはタスク・plan で指定された範囲）

`ci-check.sh` を複数同時に実行しない。完了を待ってから次を実行する。

成功ログを `coder-decisions.md` の「実行結果」に記録する。

## 構造健全性セルフチェック（必須）

- `#[path = "..."]` が 0 件であること
- 旧パス互換の wrapper `pub use` / `pub type` が 0 件であること（設計で許可された公開境界 re-export を除く）
- 旧パス参照が 0 件、または plan に明記した例外のみであること
- 同一責務の旧パス / 新パスの二重維持がないこと

セルフチェック結果を `coder-decisions.md` に記録する。
