# 統合レポート生成

Layer 1〜3 の出力を統合し、最終的なギャップ分析レポートを生成する。

## やらないこと (Do Not)
- ビルドコマンド（`cargo check` / `cargo build` / `cargo test`）を実行しないこと
- このムーブメントはレポートの統合のみを行う

## 入力

- `00-api-inventory.md`（Layer 1: API インベントリ）
- `01-interface-comparison.md`（Layer 2: インターフェース比較）
- `02-design-comparison.md`（Layer 3: 設計比較）

## 手順

### 1. サマリー集計

Layer 1〜3 の結果から以下を集計する：

| 指標 | 集計方法 |
|------|----------|
| Pekko 公開型数 | Layer 1 の Pekko 側型数 |
| fraktor-rs 公開型数 | Layer 1 の fraktor-rs 側型数 |
| カバレッジ（型単位） | Layer 2 の「同名実装 + 別名実装」/ Pekko 型数 |
| インターフェース一致率 | Layer 2 の「同名実装」のうちメソッド完全一致の割合 |
| 設計等価率 | Layer 3 の設計パターン一致数 / 比較対象数 |
| ギャップ数 | Layer 2 の「未実装」+ 「部分実装」の型数 |

### 2. カテゴリ別ギャップ統合

Layer 1 のカテゴリ分類に従い、各ギャップを統合テーブルにまとめる。
Layer 2（インターフェース）と Layer 3（設計）の情報を組み合わせて記載する。

### 3. 実装優先度の統合

Layer 3 の難易度評価に基づき、Phase 1〜4 + 対象外に分類する。
各項目に以下を付記する：
- **理由**: なぜこの Phase か
- **前提条件**: 他のギャップへの依存関係がある場合

### 4. 整合性チェック

以下の不整合がないかを確認する：
- Layer 1 で列挙された型が Layer 2 で漏れていないか
- Layer 2 で「未実装」だが Layer 3 で「設計等価」になっていないか
- サマリーの数値が本文の内容と一致しているか

不整合がある場合はレポート内に「整合性注記」として記載する。

## 出力ルール

- サマリーテーブルは必ず含める
- 各ギャップにはファイルパス・行番号の参照を含める
- Layer 間で矛盾がある場合は明示する（隠さない）

## 判定

- 統合レポートを出力完了: `統合完了`
- 入力レポートが不足・不整合: `入力不備`
- 判断不能: `判断不能`
