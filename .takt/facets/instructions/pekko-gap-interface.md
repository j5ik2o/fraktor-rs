# Layer 2: インターフェース比較

`{task}` で指定されたモジュールについて、Layer 1 のインベントリ（`00-api-inventory.md`）を入力とし、
**シグネチャレベルの対応関係** を判定する。設計評価は Layer 3 の責務。

## やらないこと (Do Not)
- ビルドコマンド（`cargo check` / `cargo build` / `cargo test`）を実行しないこと
- このムーブメントはソースコードの静的解析のみを行う

## 入力

- `00-api-inventory.md`（Layer 1 出力）

## 手順

### 1. 型の対応マッピング

Pekko側の各公開型に対し、fraktor-rs側で対応する型を特定する。

判定基準（上から優先）：

| 判定 | 条件 |
|------|------|
| 同名実装 | 名前が一致（`PascalCase` ↔ `PascalCase`） |
| 別名実装 | 名前は異なるが、責務・型パラメータ・メソッド群が同等（根拠を明記） |
| 部分実装 | 一部のメソッド/バリアントのみ存在 |
| 未実装 | 対応する型が見つからない |
| n/a | JVM固有・Akka互換層・deprecated（理由を明記） |

### 2. メソッド/バリアントの対応マッピング

対応型が見つかった場合、メソッド単位で比較する：

- **シグネチャの比較**: 引数型・戻り値型・型パラメータの対応を検証
  - Scala `implicit` → Rust ジェネリクス `TB: RuntimeToolbox` への変換は正当
  - Scala `Future[T]` → Rust `async fn → impl Future<Output=T>` への変換は正当
  - Scala `Option[T]` → Rust `Option<T>` は同等
- **可視性の比較**: `public`/`protected` ↔ `pub`/`pub(crate)` の対応
- **ライフタイム・所有権**: Scala にない Rust 固有の制約（`&self` vs `&mut self`、ライフタイムパラメータ）は注記のみ

### 3. 定数・型エイリアスの対応

コンパニオンオブジェクトの定数や型エイリアスも対象に含める。

## 出力ルール

- **対応根拠を必ず記載**: 「別名実装」「部分実装」の判定には、どの部分が対応しどの部分が欠けているかを具体的に記載
- **ファイルパスと行番号を保持**: Layer 1 から引き継いだ参照情報を維持
- **推測を混ぜない**: コードで確認できない対応関係は「不明」と記載

## 判定

- 全型の対応マッピングを出力完了: `比較完了`
- Layer 1 レポートが不足・不整合: `入力不備`
