# パッケージ移行計画

Report Directory 内の **`structure-analysis.md`**・**`structure-design.md`** およびタスク指示を根拠に、**実行可能な移行計画** を書く。

## やらないこと（Do Not）

- ソースコードの編集をしない
- ビルドを伴うコマンドを実行しない

## やること（Do）

1. **1 回の実行で扱う Phase を 1 つに限定**する（タスクに複数 Phase があっても、今回のスコープだけを計画に含める）
2. 当該 Phase について次を具体化する:
   - 移動・リネームする **ファイル／ディレクトリの一覧**（推定 diff 規模）
   - `mod` 宣言・`pub use` の更新方針（親子の再エクスポートを含む）
   - **import 更新** の検索キーワード（旧パス、`crate::` パス）
   - 影響する **テスト・クレート**（`modules/*` 以外の参照があれば列挙）
3. **実装手順** をステップに分解する（各ステップ後にコンパイルが通ることを目標にする）
4. **テスト戦略**: 構造のみの変更なら「既存テストがそのまま意味を保つ」こと、必要なら追加する境界テストを述べる
5. **リスク評価** とロールバック／分割の提案

## 既存 `plan-refactoring` との整合

- Phase 管理の原則は `plan-refactoring` と同じ: 前 Phase 未完了なら計画上で ABORT 条件を明示する

## 必須出力

- ビルトイン／プロジェクト標準の **plan** 形式に準じた `plan.md`（見出し例）:
  - 対象 Phase の概要
  - 変更ファイル一覧（作成・移動・削除）
  - ステップバイステップ手順
  - テスト戦略
  - リスク評価

## 判定（ルール評価用）

- 要件が明確で実装可能 → 本文で **実装フェーズへ進行可能** と明示
- 情報不足・スコープ曖昧 → **確認事項** を列挙し、再分析が必要と明示
