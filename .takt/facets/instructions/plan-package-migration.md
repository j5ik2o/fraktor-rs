# パッケージ移行計画

Report Directory 内の **`structure-analysis.md`**・**`structure-design.md`** およびタスク指示を根拠に、**実行可能な移行計画** を書く。

## やらないこと（Do Not）

- ソースコードの編集をしない
- ビルドを伴うコマンドを実行しない

## やること（Do）

1. **1 回の実行で扱う Phase を 1 つに限定**する（タスクに複数 Phase があっても、今回のスコープだけを計画に含める）
2. 当該 Phase について次を具体化する:
   - 移動・リネームする **ファイル／ディレクトリの一覧**（推定 diff 規模）
   - 正準パスに収束させるために **削除する旧ファイル／旧 `mod`／旧ディレクトリ**
   - `mod` 宣言・`pub use` の更新方針（公開境界のみ。旧内部パス互換は禁止）
   - **import 更新** の検索キーワード（旧パス、削除対象パス、`crate::` パス）
   - 禁止する暫定手段のチェック項目（`#[path = "..."]`、旧パス互換 `pub use`、二重維持）
   - 影響する **テスト・クレート**（`modules/*` 以外の参照があれば列挙）
3. **実装手順** をステップに分解する（各ステップ後に no-shim でコンパイルが通ることを目標にする）。その単位で無理なら、互換レイヤ追加ではなく Phase 分割を提案する
4. **テスト戦略**: 構造のみの変更なら「既存テストがそのまま意味を保つ」こと、必要なら追加する境界テストを述べる
5. **リスク評価** とロールバック／分割の提案。`#[path = "..."]` や互換 re-export に頼る案はリスク低減策として採用しない

## 既存 `plan-refactoring` との整合

- Phase 管理の原則は `plan-refactoring` と同じ: 前 Phase 未完了なら計画上で ABORT 条件を明示する
- Phase を小さく分けても no-shim で進められない場合は、設計見直しとして `design` に戻す前提を明記する
- モジュールの責務境界自体を変える必要があるなら、`knowledge/strangler-fig-migration.md` を参照し、差し替え移行（ストラングラーフィグ）に切り替える判断を plan レポートに明記する

## 必須出力

- ビルトイン／プロジェクト標準の **plan** 形式に準じた `plan.md`（見出し例）:
  - 対象 Phase の概要
  - 変更ファイル一覧（作成・移動・削除）
  - ステップバイステップ手順
  - テスト戦略
  - リスク評価
  - 正準パスに収束させるための削除対象と、禁止する暫定手段チェック

## 判定（ルール評価用）

- 要件が明確で実装可能 → 本文で **実装フェーズへ進行可能** と明示
- 情報不足・スコープ曖昧 → **確認事項** を列挙し、再分析が必要と明示
