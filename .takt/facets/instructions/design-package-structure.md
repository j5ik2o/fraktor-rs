# パッケージ構造の設計

Report Directory 内の **`structure-analysis.md`** を一次情報とし、**新しいモジュール構成案** を設計する。

## やらないこと（Do Not）

- ソースコードの編集をしない
- ビルドを伴うコマンドを実行しない

## やること（Do）

1. `structure-analysis.md` を読み、問題一覧と優先度を確認する
2. `package-design-principles` に沿い、**After のディレクトリツリー** と **モジュール間依存の向き** を提案する
3. 各責務について **正準パス（canonical path）** を 1 つ決め、廃止する旧パスを対で書く
4. **グルーピング** ごとに「なぜ同じパッケージか（CCP）」を1文で書く
5. **型の機械的複製**（例: アリティ違いの shape）がある場合は、マクロ・ジェネリクス・現状維持のいずれかを比較し、推奨と理由を書く
6. **Phase 候補** をテーブル化する（1 Phase = no-shim でコンパイル可能な単位を維持しやすい粒度）。`#[path = "..."]` や旧パス互換 `pub use` が必要になるなら、Phase をさらに細かくする
7. 許容する `pub use` は crate root / 親公開境界に限定し、それぞれが「公開 API の整形」であって「旧内部パス互換」ではないことを明記する
8. fraktor-rs 規約との整合: `core`/`std`、1 ファイル 1 公開型、Dylint、モジュール宣言方式（`mod.rs` 禁止）を崩さない

## 必須出力

- 出力契約 **structure-design** に従った `structure-design.md` を Report Directory に書く
- 分析レポートの問題番号と、設計での対応を対応表でつなぐ
- `正準パス / 廃止する旧パス / 許容する公開境界 re-export / 禁止する暫定手段` を必ず明記する

## 判定

- 設計案が具体的で `plan` に渡せる → **APPROVE**（本文で plan へ進行可能と明示）
- 分析と矛盾、循環依存を招く案、旧パス互換前提の案、規約違反 → **REJECT** と理由を書く
