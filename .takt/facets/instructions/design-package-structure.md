# パッケージ構造の設計

Report Directory 内の **`structure-analysis.md`** を一次情報とし、**新しいモジュール構成案** を設計する。

## やらないこと（Do Not）

- ソースコードの編集をしない
- ビルドを伴うコマンドを実行しない

## やること（Do）

1. `structure-analysis.md` を読み、問題一覧と優先度を確認する
2. `package-design-principles` に沿い、**After のディレクトリツリー** と **モジュール間依存の向き** を提案する
3. **グルーピング** ごとに「なぜ同じパッケージか（CCP）」を1文で書く
4. **型の機械的複製**（例: アリティ違いの shape）がある場合は、マクロ・ジェネリクス・現状維持のいずれかを比較し、推奨と理由を書く
5. **Phase 候補** をテーブル化する（1 Phase = コンパイル可能な単位を維持しやすい粒度）
6. fraktor-rs 規約との整合: `core`/`std`、1 ファイル 1 公開型、Dylint、モジュール宣言方式（`mod.rs` 禁止）を崩さない

## 必須出力

- 出力契約 **structure-design** に従った `structure-design.md` を Report Directory に書く
- 分析レポートの問題番号と、設計での対応を対応表でつなぐ

## 判定

- 設計案が具体的で `plan` に渡せる → **APPROVE**（本文で plan へ進行可能と明示）
- 分析と矛盾、循環依存を招く案、規約違反 → **REJECT** と理由を書く
