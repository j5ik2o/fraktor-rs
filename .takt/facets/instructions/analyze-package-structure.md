# パッケージ構造の現状分析

Piece Context のタスク指示に従い、**指定されたクレート／ディレクトリ** を対象に分析する。

## やらないこと（Do Not）

- ソースコードの編集をしない
- `cargo check` / `cargo build` / `cargo test` 等、ビルドを伴うコマンドを実行しない（権限・ムーブメント方針により失敗しうる）

## やること（Do）

1. タスク指示から **分析スコープ**（例: `modules/stream-core/src/core` や `modules/stream-adaptor-std/src/std`）と **1 回の実行で扱う Phase**（あれば）を確定する
2. 対象のディレクトリ構造を把握する（`Glob` / `find` 等、読み取りのみ）
3. **定量**: 子ディレクトリ数、深さの目安、明らかな「1 型 1 ディレクトリ」の乱立、類似命名のペアを数える
4. **参照実装比較**: `references/pekko` および `references/protoactor-go` の **該当領域** だけを読み、パッケージ／ディレクトリの切り方の対応を表にする  
   - リポジトリ外の他ライブラリは、タスクで明示されている場合のみ触れる
5. **問題特定**: `package-design-principles` のアンチパターン・原語に照らして優先度付きで列挙する
6. **依存の所見**: クレート境界（`{domain}-core` ↔ `{domain}-adaptor-std`）、循環の疑い、pub の粗さを文章でまとめる（完全なグラフ解析は必須ではないが、疑いは明示する）
7. **旧パス温存リスク**: 親 `mod` の広い再エクスポート、巨大 Phase、広い import 面など、`#[path = "..."]` や旧パス互換ラッパーに逃げやすい箇所を列挙する

## 必須出力

- 出力契約 **structure-analysis** に従ったレポート（`structure-analysis.md`）を Report Directory に書く
- 次ムーブメント `design` が迷わないよう、「グルーピング候補」「触れてはいけない境界」「禁止すべき暫定手段候補」を必ず残す

## 判定（ルール評価用タグ）

レポート末尾に、次のいずれかを明記する:

- 分析完了かつ次に進める → 本文で **「次: design に進行可能」** と書く
- スコープ不明・参照実装パスが特定できない等 → **REJECT** とし、確認質問を構造分析レポートに列挙する
