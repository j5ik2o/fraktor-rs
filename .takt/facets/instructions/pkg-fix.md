Piece Context に示された Report Directory 内のファイルのみ参照してください。他のレポートディレクトリは検索 / 参照しないでください。

## レポート参照方針

- Report Directory 内の最新レビューレポートを一次情報として参照してください
- 過去イテレーションのレポートは `{ファイル名}.{タイムスタンプ}` 形式で同ディレクトリに保存されています。各レポートについて `{レポート名}.*` パターンで Glob を実行し、タイムスタンプ降順で最大2件まで読み、`persists` / `reopened` の傾向を把握してから修正に入ること

## 対象レポート

- アーキテクチャレビュー: `architect-review.md`
- QAレビュー: `qa-review.md`

## やること

1. `new` / `persists` / `reopened` の指摘を `finding_id` 単位で一覧化する
2. `family_tag` がある場合は同一根本原因として束ね、ない場合は問題文と根拠から自分で根本原因を束ねる
3. 各ブロッキング指摘に対して、最小変更で修正する
4. パッケージ構造リファクタリングでは、修正後に **正準パスへ収束していること** を確認する。`#[path = "..."]`、旧パス互換 `pub use`、wrapper mod、二重維持で通してはならない
5. 修正後、`plan.md` またはタスクが指定した `./scripts/ci-check.sh` の範囲を **順番に** 実行し、結果を `coder-decisions.md` の「実行結果」に記録する
6. `coder-decisions.md` の「構造健全性セルフチェック」を更新する

## 完了条件

- 今回受け取った全ブロッキング指摘（`new` / `persists` / `reopened`）を修正したこと
- 同一根本原因の潜在箇所を同時に修正したこと
- 構造健全性セルフチェックで、禁止パターンが 0 件であること（設計で許可された公開境界 re-export を除く）
- `plan.md` またはタスクが指定した `ci-check` 範囲がすべてパスしていること

## やらないこと（重要）

- `cargo check` / `cargo build` / `cargo test` / `cargo clippy` を直接実行しないこと。必ず `./scripts/ci-check.sh` 経由で実行すること
- `ci-check.sh` を複数同時に実行しないこと（cargo ロック競合が発生する）。前のコマンドの完了を必ず待ってから次を実行すること
- `#[path = "..."]`、`include!`、旧パス互換 `pub use` / `pub type` / wrapper mod を追加しないこと
