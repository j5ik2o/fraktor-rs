テスト品質の観点から変更をレビューしてください。

## レビュー観点

- 変更差分をカバーするテストが存在するか
- 正常系だけでなく異常系・境界値・競合条件を押さえているか
- タイミング依存の待機や brittle なアサーションがないか
- テストが実装詳細に過剰結合していないか
- テスト名・前提・検証内容が読み取りやすいか

### 特に重視する点

- `sleep` や固定時間待機に依存していないか
- 非同期処理・メッセージ配送・並行処理に対して deadline 付きの待機や再試行になっているか
- 失敗時に原因が追跡しやすいメッセージになっているか
- 追加した public API や挙動変更に対応するテストがあるか

## 設計判断の参照

{report:coder-decisions.md} を確認し、テスト追加方針や意図的なスコープ制限を把握してください。
- 記録された意図的判断は FP として指摘しない
- ただし判断自体がテスト欠落を正当化できていない場合は指摘する

## 前回指摘の追跡（必須）

- まず「Previous Response」から前回の open findings を抽出する
- 各 finding に `finding_id` を付け、今回の状態を `new / persists / resolved` で判定する
- `persists` と判定する場合は、未解決である根拠（ファイル/行）を必ず示す

## 判定手順

1. 前回 open findings を抽出し、`new / persists / resolved` を仮判定する
2. 変更差分を確認し、テスト品質の観点で問題を検出する
3. ブロッキング問題（`new` または `persists`）が 1 件でもあれば REJECT とする
4. 問題がなければ APPROVE とする
