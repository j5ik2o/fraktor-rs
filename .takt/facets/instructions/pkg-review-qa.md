品質保証の観点から、**パッケージ構造リファクタリング** の変更をレビューする。

## レビュー観点（パッケージ移行向けに追加）

- **計画との一致**: `plan.md`・`structure-design.md` で決めた構成と、実際のツリー・`mod` 宣言が一致しているか
- **最終構造の実在性**: 実ファイル配置が新構造に収束しているか。親モジュールや crate root の再エクスポートだけで「移動済み」に見せていないか
- **依存方向**: `core` が `std` に依存していないか、新たな循環依存の疑いがないか
- **公開 API**: 意図しない `pub` の拡大／縮小がないか（タスクの方針に照らす）
- **互換レイヤ禁止**: `#[path = "..."]`、`include!`、旧パス互換 `pub use` / `pub type` / wrapper mod、旧パス / 新パスの二重維持がないか
- **Dylint / 規約**: 1 ファイル 1 公開型、モジュール配線、テスト配置の破壊がないか
- **テスト**: 既存テストが意味を保ち、失敗は import パス等の機械的修正で説明できるか
- **型の扱い**: マクロ化・統合で公開シンボルが意図せず変わっていないか
- **構造品質メトリクスの証跡**: `coder-decisions.md` に before/after の構造改善メトリクスが記録され、実コードと矛盾していないか
- **局所最適化の混入**: 今回の変更が「対象型だけ移した」結果に留まり、上位 dumping ground や配置予測不能性を悪化させていないか

## 構造健全性の検索（必須）

- `#\\[path\\s*=` を検索し、0件であることを確認する
- `include!` を検索し、構造隠蔽に使われていないことを確認する
- `structure-design.md` で廃止対象になっている旧パスの参照残りを検索する
- `pub use` は、`structure-design.md` で許可された公開境界だけに残っているか確認する

## 一般 QA 観点

- テストカバレッジと品質、エラーハンドリング、保守性（`refactoring-review-qa` と同様の観点を、スコープが構造変更に限定されるよう読み替える）

## 設計判断の参照

`coder-decisions.md` を確認する。記録された意図的な判断は FP として指摘しないが、判断自体が不適切なら指摘する。

## ナレッジ

`package-design-principles` のチェックリストを参照し、構造面の REJECT 条件を明確にする。

## やらないこと

`cargo` 系コマンドを実行しない。ビルド検証は implement / fix の責務。

## 判定

- ブロッキング問題なし → **approved**
- ブロッキングあり → **needs_fix**（具体的なファイルパスと修正方針）

次のいずれかがあれば **必ず blocking** とする:

- `#[path = "..."]` または構造隠蔽の `include!`
- 旧パス互換の `pub use` / `pub type` / wrapper mod
- 同一責務の旧パス / 新パスの二重維持
- 実ファイル移動なし、または import 未更新のまま見かけだけ新構造にしている状態
- `coder-decisions.md` に構造改善メトリクスがなく、改善量が検証不能
