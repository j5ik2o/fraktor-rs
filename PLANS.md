# PLANS

プロジェクト全体の意思決定・進行管理を共有するための基本ドキュメントです。詳細な設計判断は `.kiro/steering/` や各 spec へリンクしてください。以下はテンプレート項目なので、必要に応じて追記・削除して運用します。

## 1. 目的と位置付け
- **ゴール**: プロジェクトが短期・中期で達成したい成果やプロダクト価値を要約する。
- **適用範囲**: 開発対象、除外する領域、依存する外部要素を明確にする。

## 2. 優先テーマとマイルストーン
| 優先度 | テーマ | 目的 | 担当/オーナー | 予定完了日 | ステータス |
| --- | --- | --- | --- | --- | --- |
| P0 | 例: Queue Front API 導入 | 例: backend 非公開化とカテゴリ enum 統一 | 例: Runtime チーム | 2025-12-01 | In Progress |
| P1 |  |  |  |  |  |

> ※マイルストーンは `scripts/ci-check.sh all` を通過することを基本完了条件とするなど、客観的な Definition of Done を添付。

## 3. 重要設計判断 (Key Design Decisions)
| 日付 | 判断内容 | 背景/根拠 | 影響範囲 | 詳細リンク |
| --- | --- | --- | --- | --- |
| 2025-11-12 | 例: キュー backend 直接公開禁止 | 例: 一貫性欠如と lint 違反が続出したため | utils-core, actor-core | `.kiro/steering/tech.md#queue-facade` |

記載ルール:
1. 重大な方針は 1 行に 1 判断で記載し、根拠と参照ファイルを必ず紐付ける。
2. 過去判断が失効した場合は「Status」欄を追加して `Deprecated` 等へ更新。

## 4. 進行状況トラッキング
- **現在のスプリント**: 期間、テーマ、完了条件を箇条書きで記載。
- **リスク/ブロッカー**: 発生中の課題、影響、対策オーナーを列挙。
- **メトリクス**: 成功指標（例: CI パス率、ターゲット別ビルド状況、ベンチマーク値）を定期更新。

## 5. spec / steering との連動
- Active Specs: `/prompts:kiro-spec-status` の結果サマリを転記。
- Steering 更新点: `.kiro/steering/product.md` / `tech.md` / `structure.md` への追加・修正をここでも簡潔に共有。
- 参照ガイド: `docs/guides/*.md` にある作業手順へのリンク。

## 6. 次のアクション
1. 優先テーマごとに直近のタスク（1〜3 件）を列挙し、担当・予定日を紐付ける。
2. プロジェクト横断で決めるべき事項（例: 新 Lint 도입、ターゲット追加）を「要議論事項」としてまとめる。

---
このテンプレートをベースに、必要なセクションを埋めながら運用してください。更新時はコミットメッセージに「plans」タグを入れるなど、追跡しやすいワークフローにすると便利です。
