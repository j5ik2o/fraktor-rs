# 目的
曖昧なサフィックスを避け、責務が明確に伝わる命名へ導く。

# 基本ルール
- `Manager`/`Util`/`Facade`/`Service`/`Runtime` を新規命名で使用しない
- 既存名に該当サフィックスが含まれる場合は、責務に沿った具体名へ置換案を提示する
- 既存命名規約（`*Shared`/`*Handle` など）と矛盾しないことを確認する

# 手順
1. 変更対象の公開 API・型・トレイト・モジュール名を洗い出す  
2. 禁止サフィックスの該当有無を確認する  
3. 該当する場合は責務を一文で要約する  
4. 要約に基づき具体名へ置換案を作る  
5. 既存の命名パターンと整合するかを確認する  
6. 置換が確定したら、参照箇所・ドキュメント・spec を更新する  

# 置換の考え方
- 「何をするか」ではなく「何であるか」を表す名詞を優先する  
- 具体的な責務が複数ある場合は、責務分割や構成名（例: `*Coordinator`/`*Router`）を検討する  
- `Shared`/`Handle` の使い分けは既存ルールに従う  

# 命名候補の例
- データ保持中心: `*Table` / `*Registry` / `*Catalog` / `*Index`  
- 選択や分岐: `*Router` / `*Selector` / `*Policy`  
- 仲介・制御: `*Coordinator` / `*Dispatcher` / `*Controller`  
- アダプタ・変換: `*Adapter` / `*Bridge` / `*Mapper`  

# 迷ったときの確認事項
- 「所有とライフサイクル」を持つか  
- 「単一責務」に落とし込めるか  
- 既存のドメイン語彙と一致しているか  

# 注意
- 既存の外部 API 名や第三者ライブラリ名は無理に改名しない  
- 既存コードの広範囲なリネームは、必ず利用者の意図確認を行う
