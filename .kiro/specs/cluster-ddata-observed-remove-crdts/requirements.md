# 要件定義

## はじめに

cluster の Distributed Data 利用者は、集合・マップ形の収束データ（複数ノードで並行に追加・削除・更新しても最終的に同一状態へ収束する観測除去型のコレクション）を扱えない。現状の `ddata`（`modules/cluster-core-kernel/src/ddata/`）には scalar 系 CRDT（`Flag` / `GCounter` / `PNCounter` / `PNCounterMap`）と因果クロック（`VersionVector`）・最終書き込み優先レジスタ（`LWWRegister`）しかなく、Pekko parity で要求される observed-remove set / map 系が欠落している（gap analysis カテゴリ9 の未対応項目）。

本仕様は、既に実装済みの CRDT 基盤の上に、Replicator runtime に依存しない pure CRDT データ型として `ORSet` / `ORMap` / `ORMultiMap` / `LWWMap` の 4 型を追加する。収束規則は Pekko の distributed-data 実装（`references/pekko/distributed-data/.../ORSet.scala` / `ORMap.scala` / `ORMultiMap.scala` / `LWWMap.scala`）に準拠する。

## 境界コンテキスト

- **対象範囲**: `ORSet`（add-wins 観測除去集合）、`ORMap`（キーは観測除去・値は CRDT 再帰併合）、`ORMultiMap`（多値バインディング）、`LWWMap`（キー単位 last-writer-wins）の利用者から観測可能な振る舞い（公開操作・並行競合の収束結果・観測除去・退役ノードの pruning・差分と全状態併合の一致）。CRDT 則（可換・結合・冪等）の検証。no_std ビルドと構造制約・イミュータブル制約の遵守。
- **対象外**: `Replicator` / `ReplicatorSettings` 実行体（gossip ベース複製、write/read repair、差分伝播ループ）、`DistributedData` extension（classic / typed）、`ReplicatorMessageAdapter`、`DurableStore` SPI と durable store std adapter、typed 層・std 層への新規 API 追加。Replicator protocol core（`Get` / `Update` / `Delete` / `Subscribe`）への結線は、protocol 側が generic に `ReplicatedData` を扱うため、新型がその契約に適合することの確認に留める。
- **隣接システム／スペックへの期待**: 既存 ddata 基盤（CRDT 基底 SPI `ReplicatedData` / `DeltaReplicatedData` / `RemovedNodePruning`、`VersionVector`、`LWWRegister`、`SelfUniqueAddress`、`Key`）が公開契約として再利用可能であること。本仕様はこれらを再定義せず依存する。Replicator protocol core が新型を扱う際は、generic な `ReplicatedData` 適合のみを前提とする。

## 要件

### 要件 1: ORSet（観測除去集合 / add-wins）

**目的:** Distributed Data 利用者として、複数ノードで並行に要素を追加・削除しても収束する集合を扱うために、観測除去セマンティクスを持つ `ORSet` が欲しい

#### 受け入れ基準

1. 利用者が自ノード識別とともに要素を追加したとき、`ORSet` はその要素を含む新しいインスタンスを返さなければならない
2. 利用者が要素を削除したとき、`ORSet` は削除時点で観測済みの追加のみを取り消した新しいインスタンスを返さなければならない
3. 同一要素に対する追加と削除が並行して起きた場合、`ORSet` は併合後にその要素を保持しなければならない（add-wins）
4. 削除後に同一要素が再追加された場合、`ORSet` は併合後にその要素を保持しなければならない
5. `ORSet` は常に、含有判定と要素列挙を提供しなければならない
6. `ORSet` は常に、2 つの値を併合するとき併合の順序・回数によらず同一の要素集合へ収束しなければならない

### 要件 2: ORMap（観測除去キー付きマップ / 値 CRDT 再帰併合）

**目的:** Distributed Data 利用者として、キーごとに CRDT 値を保持し並行更新が収束するマップを扱うために、`ORMap` が欲しい

#### 受け入れ基準

1. 利用者が自ノード識別・キー・値とともに値を設定したとき、`ORMap` はそのキーに値を関連付けた新しいインスタンスを返さなければならない
2. 同一キーへの並行な値更新が起きた場合、`ORMap` は併合後にそのキーの値を、値 CRDT の併合規則で収束させなければならない
3. 利用者がキーを削除し、かつ他ノードで同一キーへの並行更新が起きた場合、`ORMap` は併合後にそのキーを保持しなければならない（キー集合は add-wins）
4. 利用者が更新関数によってキーの値を更新するとき、`ORMap` は既存値（存在しなければ呼び出し元が与えた初期値）に更新関数を適用した新しいインスタンスを返さなければならない
5. 値の置換（put）は値の因果履歴を保持しないため、集合値のマップが必要な場合、利用者は `ORMultiMap` を使わなければならない
6. `ORMap` は常に、キーによる値取得とエントリ列挙を提供しなければならない
7. `ORMap` は常に、2 つの値を併合するとき併合の順序・回数によらず同一のキー・値へ収束しなければならない

### 要件 3: ORMultiMap（多値バインディング）

**目的:** Distributed Data 利用者として、1 つのキーに複数の値を関連付け、並行な追加・削除が収束する多値マップを扱うために、`ORMultiMap` が欲しい

#### 受け入れ基準

1. 利用者が自ノード識別・キー・要素とともにバインディングを追加したとき、`ORMultiMap` はそのキーの値集合に要素を加えた新しいインスタンスを返さなければならない
2. 利用者がキーから要素を削除し、その結果として値集合が空になった場合、`ORMultiMap` は可視エントリからそのキーを除いた新しいインスタンスを返さなければならない
3. 同一キーの同一要素に対する追加と削除が並行して起きた場合、`ORMultiMap` は併合後にその要素を保持しなければならない（値集合は add-wins）
4. `ORMultiMap` は常に、キーによる値集合取得とエントリ列挙を提供しなければならない
5. `ORMultiMap` は常に、2 つの値を併合するとき併合の順序・回数によらず同一のキー・値集合へ収束しなければならない

### 要件 4: LWWMap（キー単位 last-writer-wins マップ）

**目的:** Distributed Data 利用者として、キーごとに最終書き込み優先で値を保持し並行設定が決定的に収束するマップを扱うために、`LWWMap` が欲しい

#### 受け入れ基準

1. 利用者が自ノード識別・キー・値・タイムスタンプとともに値を設定したとき、`LWWMap` はそのキーに値とタイムスタンプを関連付けた新しいインスタンスを返さなければならない
2. 同一キーへの並行な値設定が起きた場合、`LWWMap` は併合後にタイムスタンプが大きい方の値を保持しなければならない
3. 同一キーへの並行な値設定が起き、かつタイムスタンプが等しい場合、`LWWMap` は併合後に `UniqueAddress` 順序で小さいノードの値を保持しなければならない
4. 利用者がキーを削除し、かつ他ノードで同一キーへの並行設定が起きた場合、`LWWMap` は併合後にそのキーを保持しなければならない（キー集合は add-wins）
5. `LWWMap` は常に、値の最終書き込み判定に既存 `LWWRegister` の clock 契約（呼び出し元が与えるタイムスタンプ）を用いなければならない
6. `LWWMap` は常に、キーによる値取得とエントリ列挙を提供しなければならない
7. `LWWMap` は常に、2 つの値を併合するとき併合の順序・回数によらず同一のキー・値へ収束しなければならない

### 要件 5: CRDT 基底契約への適合・差分伝播・退役ノード整理

**目的:** 将来の Replicator や利用者が 4 型を一様に扱えるよう、各型が既存 CRDT 基底契約に適合し、差分伝播と退役ノード整理が正しく動作してほしい

#### 受け入れ基準

1. `ORSet` / `ORMap` / `ORMultiMap` / `LWWMap` は常に、状態ベース併合契約（`ReplicatedData`）に適合しなければならない
2. 差分伝播を提供する場合、各型は差分の適用結果を、同じ更新を全状態併合した結果と一致させなければならない
3. ノードの退役が確定したとき、各型はそのノードに由来する因果情報を整理し、退役前に観測済みの状態を保存した新しいインスタンスを返さなければならない
4. 差分伝播を提供する場合、各型は全状態併合のみでも収束しなければならない（差分配送は因果順序を前提としてよい）

### 要件 6: no_std・構造・イミュータブル制約（保守者・運用者観点）

**目的:** 保守者として、既存 ddata 資産と一貫し no_std 環境でビルドできるよう、各型が構造・イミュータブル制約を満たしてほしい

#### 受け入れ基準

1. 各型の更新操作は常に、内部可変性を用いず、元のインスタンスを変更しない新しいインスタンスを返さなければならない
2. `ddata` モジュールは常に、no_std ビルドと構造 lint（1 ファイル 1 公開型・`mod.rs` 禁止・sibling `*_test.rs` 配置・親モジュールでのバレル再エクスポート禁止）を通過しなければならない
3. 各型は常に、併合の可換律・結合律・冪等律を満たさなければならない
4. 各型は常に、既存の CRDT 基底 SPI・`VersionVector`・`LWWRegister`・`SelfUniqueAddress`・`Key` を再利用し、同等の基盤型を新設してはならない
